# Story 16.4: Server Launch and Connection Details

**Status:** review
**Epic:** 16 - Personal Runs - Private User-Created Archipelago Games
**Date:** 2026-05-12

## Story

As a run owner,
I want to start the Archipelago server for my personal run and see connection details,
So that participants can connect and the game can begin.

## Acceptance Criteria

1. `POST /api/v1/runs/{runId}/start` (owner only, run `draft` with ≥1 game configured):
   - Generates `connectionPassword = bin2hex(random_bytes(8))`, stores it on the run record.
   - Transitions run status to `starting`.
   - Dispatches `LaunchPersonalRunJob` via Messenger.
   - Response: **202** `{ data: { runId, status: "starting" } }`.

2. Internal callback `POST /api/v1/runs/{runId}/running` (bridge-token auth, sent by runner once container is ready):
   - Payload: `{ connectionHost, connectionPort }`.
   - Transitions status `starting` → `active`, stores `connection_host` and `connection_port`.
   - `connectionPassword` already set at start time (AC 1); not resent in callback.

3. `GET /api/v1/runs/{runId}` payload includes `connectionHost`, `connectionPort`, `connectionPassword` when status is `active`; all three are `null` otherwise.

4. `POST /api/v1/runs/{runId}/start` when status is already `starting` or `active` → 422 `run_already_active`.

5. `POST /api/v1/runs/{runId}/stop` (owner only, run `active`):
   - Transitions status to `stopping`.
   - Dispatches `StopPersonalRunJob` via Messenger.
   - Response: **202** `{ data: { runId, status: "stopping" } }`.

6. Internal callback `POST /api/v1/runs/{runId}/stopped` (bridge-token auth):
   - Transitions status `stopping` → `idle`.
   - Clears `connection_host`, `connection_port`, `connection_password`.

## Dev Notes

- Both `start` and `stop` are **async (202)**. The frontend must poll `GET /api/v1/runs/{runId}` (or use SSE) to detect the transition to `active` or `idle`.
- Poll cadence in frontend: 3s while status is `starting` or `stopping`, stop polling when status reaches `active`, `idle`, or a terminal state.
- Status lifecycle summary: `draft` → `starting` → `active` → `stopping` → `idle` (repeatable). `completed` and `cancelled` are terminal.
- The runner must receive `personal_run_id` (not an event `session_id`) in the job payload; callback URLs are `/api/v1/runs/{runId}/running` and `/api/v1/runs/{runId}/stopped`.

### Completion Notes List

- `PersonalRunLifecycle` is a dedicated service (not mixed into `PersonalRunDrafts`) - cleaner separation as it needs `MessageBusInterface`.
- `LaunchPersonalRunJob` / `StopPersonalRunJob` are minimal message classes routed to the existing `run_server` transport; no handler is implemented in this story (the runner process will handle them out of band).
- `connectionPassword` is generated by `PersonalRun::start()` and stored at dispatch time (not at callback time). The callback `/running` only stores host+port.
- AC 3 nulling: connection fields are masked to `null` in `PersonalRunDrafts::payload()` when status ≠ `active`, so the password is never leaked during `starting` status.
- `session_id` field added to migration and entity (with getter) per task spec; not used in any v1 AC but reserved for future runner integration.
- PHPStan fix: transport filter narrows to `object`, solved by `assertInstanceOf` on the extracted message before accessing the public property.
