# Story 23.1: Weekly Run Domain Model & Bounded Context

## Story

**As a** developer,
**I want** a `WeeklyRuns` bounded context with `WeeklyTemplate`, `WeeklyRun`, and `WeeklyEntry` domain entities and a `WeeklyRunnerGatewayInterface`,
**So that** the weekly run lifecycle is cleanly encapsulated with no coupling to other bounded contexts.

## Dev Notes

### Architecture: individual runs per player

Each player generates their own personal Archipelago game on demand - there is **no shared multiworld session**. The key insight: Archipelago's `--seed` flag makes all independently generated games identical in layout. Every player who launches their weekly run gets the same randomized world, but each runs their own server.

`WeeklyRun` is a lightweight "week record": it holds the seed and date window. It owns no runner session. `WeeklyEntry` owns the runner session (`externalSessionId`) for one player's one attempt.

### WeeklyEntry.externalSessionId

This is the runner-side session ID (same kind of ID as `PersonalRuns` uses for a `Run`). It is `null` until the player clicks "Lancer ma partie". Set by `WeeklyEntry::launch(externalSessionId, launchedAt)`.

### Template YAML player name

The admin configures the YAML using `yaml-option-editor.tsx`. The `name` field in the YAML is set to "ArchiLAN" (or whatever the admin enters) - this is just a placeholder. When the player launches their run, the Application service substitutes their actual `displayName` before calling `WeeklyRunnerGatewayInterface::launchEntry()`. The stored `yamlConfig` always keeps the placeholder name; name substitution happens at runtime.

### completionTimeSeconds reference point

`completionTimeSeconds = goalReachedAt - run.startedAt` (not entry.launchedAt). The `startedAt` of the `WeeklyRun` is set Monday 00:00 when the run record is created. This ensures the leaderboard compares all players on the same baseline - someone who launches Monday morning and someone who launches Wednesday are compared against the same Monday 00:00 reference.

### Cross-context coupling rules

`WeeklyRuns` **never** imports from `PersonalRuns`, `Sessions`, `Events`, or `Identity`. References to other contexts are plain string UUIDs:
- `WeeklyTemplate.gameId` - UUID of a `GameSelection\Domain\Game`. No `#[ORM\ManyToOne]`.
- `WeeklyEntry.userId` - UUID of an `Identity\Domain\User`. No `#[ORM\ManyToOne]`.

DBAL joins in Application/Query layer handle cross-context reads.

### NullWeeklyRunnerGateway

`launchEntry()` returns `['externalSessionId' => 'null-session-id', 'connectionInfo' => ['host' => 'localhost', 'port' => 38281, 'password' => null]]`. `terminate()` is a no-op. `getStats()` returns `['checksTotal' => 0, 'itemsTotal' => 0, 'goalReachedAt' => null]`.
